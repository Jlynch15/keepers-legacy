Navigation Shell Architecture

How should the Godot scene tree organize the sidebar, screens, and transitions? Pick the approach that makes sense to you.

A

Single Scene + Swappable Content

One main scene holds the sidebar permanently. Screen content is loaded/unloaded into a container node. Simple, flat structure.

Recommended
MainScene.tscn ├── Sidebar (always visible, handles nav clicks) ├── ContentContainer (swaps child scenes) │ └── [current screen] (HabitatFloor / Shop / Orders / ...) ├── TransitionOverlay (crossfade + fade-to-black layers) └── ImmersiveLayer (story events, level-up — above everything)

Pros

  • Sidebar never rebuilds — state preserved
  • Clear ownership: MainScene manages all transitions
  • Easy to reason about — one scene tree
  • Screens are independent .tscn files

Cons

  • All screens share one tree — careful with signals
  • Memory: previous screen fully unloaded
B

Layered Scenes with Visibility Toggle

All screens are loaded at startup and hidden/shown. Faster switching since nothing is loaded/unloaded, but uses more memory.

MainScene.tscn ├── Sidebar ├── Screens │ ├── HabitatFloor (visible: true) │ ├── Shop (visible: false) │ ├── Orders (visible: false) │ ├── Breeding (visible: false) │ ├── Pedia (visible: false) │ └── Settings (visible: false) ├── TransitionOverlay └── ImmersiveLayer

Pros

  • Instant transitions — no load time
  • Screens keep their scroll position / state
  • Simple toggle logic

Cons

  • All screens in memory at once (~6 screens)
  • Hidden screens may still process (need to pause)
  • Harder to manage as screens get complex
C

Godot SceneTree.change_scene — Full Swap

Each screen is a completely independent scene including its own sidebar. Godot swaps the entire scene tree on navigation.

HabitatFloorScene.tscn (independent, has own sidebar copy) → change_scene → ShopScene.tscn (independent, has own sidebar copy) → change_scene → OrdersScene.tscn (independent, has own sidebar copy)

Pros

  • Screens fully independent — no shared state
  • Clean memory per screen

Cons

  • Sidebar duplicated in every scene
  • Sidebar state (active button) must sync manually
  • Crossfade transitions harder to implement
  • More files, more maintenance
My recommendation: Option A gives us the cleanest architecture. The sidebar lives once, transitions are centralized, and each screen is its own .tscn file that can be built independently. The crossfade and fade-to-black effects sit on a dedicated overlay layer. This maps directly to how the HTML mockups are structured.
